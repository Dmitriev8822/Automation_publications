# Модуль telegram

## Назначение

`app.telegram` публикует уже сгенерированный текст поста и опциональное изображение в Telegram-канал через `pyTelegramBotAPI`. Также модуль регистрирует кнопку ручной публикации в чате с ботом.

Модуль не генерирует контент, не ищет новости и не записывает статус публикации в БД. При нажатии кнопки он вызывает переданный callback, а сам бизнес-сценарий остается в `app.service`.

## Публичные классы и функции

- `TelegramPublisher` — адаптер публикации в Telegram.
- `TelegramPublisher.publish_post(post: GeneratedPost, image: ImageAsset | None = None) -> int` — публикует текстовый пост или изображение с подписью и возвращает `telegram_message_id`.
- `TelegramPublisher.register_manual_publish_handler(publish_callback)` — регистрирует `/start` и кнопку `📰 Опубликовать новость`; callback получает функцию отправки коротких progress-сообщений пользователю.
- `TelegramPublisher.start_manual_polling()` — запускает long polling для обработки команд и кнопки ручной публикации.

## Настройки

Модуль использует настройки из `app.config.Settings`:

| Переменная | Назначение |
| --- | --- |
| `TELEGRAM_BOT_TOKEN` | токен бота, полученный у BotFather; формат `<bot_id>:<secret>`, например `123456789:AA...` |
| `TELEGRAM_CHANNEL_ID` | идентификатор канала, куда бот публикует новости: публичный username вида `@channel_name` или внутренний id канала вида `-1001234567890` |

Если одна из настроек отсутствует, `TelegramPublisher` выбрасывает `ValueError` с понятным сообщением.

## Как создать бота

1. Откройте Telegram и найдите `@BotFather`.
2. Выполните команду `/newbot`.
3. Укажите имя и username бота.
4. Сохраните выданный токен в переменную `TELEGRAM_BOT_TOKEN` целиком: сначала идут цифры bot id, затем двоеточие и секретная часть, например `123456789:AA...`. Не добавляйте кавычки, пробелы, слово `bot` или username бота.

## Как добавить бота администратором канала

1. Откройте настройки Telegram-канала.
2. Перейдите в раздел администраторов.
3. Добавьте созданного бота.
4. Выдайте боту право публиковать сообщения.
5. Укажите канал в `TELEGRAM_CHANNEL_ID`, например `@my_channel` для публичного канала. Для приватного канала используйте внутренний id канала вида `-1001234567890`; это именно id канала/чата для публикации, а не bot id из токена и не username пользователя.

## Как работает публикация

- Если `image` не передан, `publish_post()` вызывает `send_message(chat_id=..., text=post.text)`.
- Если `image` передан и текст помещается в лимит подписи Telegram, `publish_post()` вызывает `send_photo(chat_id=..., photo=..., caption=post.text)`. Если текст длиннее 1024 символов, модуль сначала отправляет изображение без подписи, а затем публикует текст отдельным `send_message`, потому что Telegram не принимает длинные captions у фото.
- Для `ImageAsset.data` модуль задает multipart filename с расширением по `mime_type` (`.png`, `.jpg`, `.webp`), чтобы Telegram корректно распознавал формат сгенерированного файла.
- Если Telegram возвращает `Bad Request: IMAGE_PROCESS_FAILED`, модуль логирует предупреждение и повторяет публикацию как текстовый пост через `send_message`, чтобы не срывать весь бизнес-процесс из-за непригодного изображения.
- Для `ImageAsset` поддерживаются `data`, `file_path` и `url`.
- Метод возвращает `message.message_id` из ответа Telegram.
- Ошибки Telegram API при публикации оборачиваются в `RuntimeError` с префиксом `Telegram publication failed`, кроме `IMAGE_PROCESS_FAILED` для изображения: в этом случае выполняется text-only fallback. Ошибка `401 Unauthorized` при запуске polling оборачивается в понятное сообщение с подсказкой проверить `TELEGRAM_BOT_TOKEN`.

## Проверка токена и channel id

Если при старте polling появляется ошибка `401 Unauthorized`, Telegram не принял `TELEGRAM_BOT_TOKEN`. Нужно заново скопировать токен у `@BotFather` командой `/token` или перевыпустить его командой `/revoke`, затем обновить `.env` и перезапустить приложение. Для быстрой проверки только Telegram-токена можно выполнить `python app/main.py --check-telegram`: команда делает `getMe`, показывает имя подключенного бота при успехе и не запускает scheduler.

`TELEGRAM_CHANNEL_ID` не используется для получения кнопки в личном чате с ботом: кнопка появляется после `/start`, если polling успешно запущен с правильным токеном. `TELEGRAM_CHANNEL_ID` нужен позже, когда сервис публикует готовую новость в канал. Бот должен быть администратором этого канала с правом публикации сообщений.

## Ручная публикация по кнопке

`register_manual_publish_handler()` добавляет `/start`, `/menu`, главное меню и сценарий согласования новости:

1. пользователь нажимает `📰 Опубликовать новость`;
2. Telegram-модуль вызывает callback подготовки черновика из `app.service`;
3. бот показывает текст, источник и наличие картинки;
4. пользователь выбирает `✅ Принять`, `❌ Отменить`, `✍️ Перегенерировать текст` или `🖼️ Перегенерировать картинку`;
5. только после `✅ Принять` вызывается callback публикации, который сохраняет запись в БД и отправляет пост в Telegram-канал.

Для совместимости метод также поддерживает legacy-вызов с одним callback: в этом режиме поведение остается прежним и публикация выполняется сразу.

## Тестирование без реального Telegram

Unit-тесты должны передавать fake/mock bot в `TelegramPublisher(settings=..., bot=...)`. Такой бот реализует методы `send_message()`, `send_photo()`, `message_handler()` и `infinity_polling()` и возвращает объект с `message_id`. Это позволяет проверить публикацию текста, изображений, text-only fallback при `IMAGE_PROCESS_FAILED`, возврат ID, регистрацию кнопки, отправку progress-сообщений и проброс ошибок без реальных HTTP-запросов.

Запуск тестов модуля:

```bash
pytest tests/test_telegram.py
```

## Консольное логирование

Модуль пишет INFO-логи при создании `TelegramPublisher`, регистрации ручных обработчиков, запуске polling, проверке токена через `getMe`, начале публикации поста и успешном получении `message_id`. Секретный токен бота в логи не выводится.

## Диалог контент-плана

`register_content_plan_handler(generate_callback, approve_callback, list_callback=None)` добавляет кнопку `🗓️ Контент план`. Если передан `list_callback`, сначала показывается подменю `👀 Посмотреть КП` / `📝 Составить КП`; просмотр выводит краткий список запланированных пунктов, а составление запускает сценарий:

1. пользователь нажимает кнопку;
2. бот просит описать контент-план в свободном формате;
3. `generate_callback(description)` возвращает `ContentPlan`; если генерация падает из-за OpenRouter или другой ошибки, бот отправляет пользователю понятное сообщение и не пробрасывает исключение наружу, чтобы long polling не остановился;
4. бот показывает структурированный план и кнопки `🔄 Перегенерировать` / `✅ Согласовать`;
5. перегенерация повторно вызывает `generate_callback` с тем же описанием;
6. согласование вызывает `approve_callback(plan)` и сообщает, что план сохранен и будет выполнен по расписанию.

Модуль только ведет Telegram-диалог: генерация плана остается в `app.ai`, сохранение — в `app.database`, выполнение — в `app.service`.

## Продолжение согласования контент-плана

После первой генерации контент-плана пользователь может не только нажать `🔄 Перегенерировать`, но и отправить обычное текстовое уточнение. `TelegramPublisher` сохраняет историю правок в памяти диалога и передает ее в `generate_callback(description, dialog_context)`, если callback поддерживает второй аргумент. Благодаря этому AI учитывает исходное описание, предыдущие варианты и новые комментарии пользователя до тех пор, пока пользователь не нажмет `✅ Согласовать`.

Кнопки главного меню не считаются обычными уточнениями внутри активного диалога контент-плана. Если пользователь во время такого диалога нажимает `⏰ Напоминания`, незавершенный диалог контент-плана сбрасывается и запускается настройка напоминаний, а не повторная генерация плана.


## Навигационные кнопки

В диалогах, где пользователь может зависнуть на промежуточном шаге, модуль показывает дополнительные кнопки управления:

- `⬅️ Назад` — в согласовании контент-плана возвращает пользователя к шагу свободного описания и очищает текущий черновик плана;
- `🏠 Меню` — закрывает активный диалог контент-плана, настройки напоминаний или карточку ожидающего решения поста и возвращает клавиатуру главного меню;
- `❌ Отмена` — работает как безопасное закрытие текущего диалога. Для напоминания перед публикацией отмена закрывает только пользовательское решение в чате и не меняет запланированный пункт; чтобы отменить саму публикацию, нужно нажать `❌ Не выкладывать`.

Эти кнопки не запускают бизнес-логику публикации, генерации или сохранения сами по себе: они только управляют состоянием in-memory диалогов `TelegramPublisher` и возвращают пользователю понятное меню.

## Напоминания и согласование публикации

Кнопка `⏰ Напоминания` запускает диалог постоянной настройки: пользователь вводит положительное число минут до публикации или `0`/`отключить`, чтобы выключить уведомления. Значение сохраняется в `reminder_minutes_before`, а чат для уведомлений — в `reminder_chat_id`; `app.main` дополнительно сохраняет настройку в БД через `ReminderSettingsRepository`. Включенная настройка применяется ко всем еще запланированным пунктам: и к уже сохраненным, и к новым контент-планам.

Кнопки главного меню и кнопки согласования контент-плана не обрабатываются как ответ с количеством минут внутри активного диалога напоминаний. Если пользователь во время настройки напоминаний нажимает `🗓️ Контент план`, диалог напоминаний сбрасывается и начинается сценарий контент-плана. Это предотвращает ситуацию, когда нажатие `⏰ Напоминания` или `🗓️ Контент план` попадает в чужой обработчик из-за уже открытого диалога.

Перед публикацией `send_publication_reminder(chat_id, item_id, item)` отправляет пост, описание картинки и клавиатуру:

- `✅ Одобрить пост` — вызывает callback немедленной публикации поста с картинкой;
- `❌ Не выкладывать` — вызывает callback отмены пункта;
- `✍️ Перегенерировать текст` — вызывает callback перегенерации текста и повторно показывает обновленный пост;
- `🖼️ Перегенерировать картинку` — вызывает callback перегенерации `image_prompt` и повторно показывает обновленный пост.

Callback-и регистрируются через `register_publication_approval_handler(...)`; модуль Telegram только ведет диалог, а публикация, отмена и перегенерация остаются в `app.service`, `app.ai` и `app.database`. Для ожидающих решения постов ключ чата нормализуется в строку: это важно, потому что сохраненный в БД `chat_id` возвращается строкой, а входящие Telegram updates обычно содержат числовой `chat.id`. Благодаря нормализации кнопки согласования, отмены и перегенерации находят один и тот же ожидающий пост и не отвечают ошибочно `Нет поста, ожидающего решения.`.
