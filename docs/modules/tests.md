# Модуль тестирования

## Назначение

Набор тестов проверяет основные модули приложения и рассчитан на быстрый запуск без настоящих внешних API, сетевых запросов и production-БД. Команда по умолчанию запускает только быстрые проверки:

```bash
pytest
```

## Настройка pytest

Конфигурация находится в `pytest.ini`:

- `pythonpath = .` добавляет корень проекта в путь импортов;
- `testpaths = tests` ограничивает поиск тестов директорией `tests/`;
- `addopts = -m "not integration"` исключает потенциально медленные integration-тесты из обычного запуска;
- маркеры `unit` и `integration` разделяют быстрые проверки и проверки с внешними зависимостями.

Файл `tests/conftest.py` автоматически помечает все тесты без маркера `integration` как `unit`. Поэтому существующие быстрые тесты не зависят от порядка запуска и выполняются командой `pytest`.

## Какие тесты есть

- `tests/test_schemas.py` — проверяет Pydantic-схемы и ошибки валидации для новостей, сгенерированных постов, изображений и опубликованных постов.
- `tests/test_config.py` — проверяет значения по умолчанию, переопределение переменными окружения, допустимые окружения `dev`/`test`/`prod` и обязательность секретов только для production.
- `tests/test_database.py` — проверяет создание таблицы постов, сохранение сгенерированных постов, защиту от дублей по `source_url`, получение записей и переходы статусов `published`/`failed`.
- `tests/test_ai.py` — проверяет AI-клиент на fake HTTP-ответах: поиск новостей, генерацию текста поста, обработку пустого или некорректного JSON и генерацию изображения без обращения к OpenRouter.
- `tests/test_telegram.py` — проверяет публикацию текста и изображений через fake bot, обработку ошибок Telegram и обязательные настройки канала.
- `tests/test_service.py` — проверяет полный сервисный сценарий: поиск свежей новости, пропуск дублей, генерацию поста и изображения, публикацию и отметку ошибок.
- `tests/test_scheduler.py` — проверяет создание scheduler, регистрацию job-функции, логирование ошибок job, безопасный импорт `app.main` и отказ от старта scheduler при провале startup-тестов.

## Fake/mock-объекты

- `FakeHTTPClient` и `FakeResponse` в `tests/test_ai.py` заменяют HTTP-клиент OpenRouter и возвращают заранее заданные JSON-ответы.
- `FakeBot` в `tests/test_telegram.py` заменяет Telegram bot API и сохраняет отправленные сообщения/фото в памяти.
- `FakeAIClient`, `FakeTelegramPublisher` и `FakeRepository` в `tests/test_service.py` проверяют orchestration-логику сервиса без внешних API и без реальной БД.
- В `tests/test_database.py` используется временная SQLite in-memory БД или временный путь из `tmp_path`, поэтому production-БД не затрагивается.
- В `tests/test_scheduler.py` применяется `monkeypatch`, чтобы импорт и `main()` не запускали бесконечный цикл или реальный scheduler при неуспешных startup-тестах.

## Как запускать

Быстрый набор без integration-тестов:

```bash
pytest
```

Только явно быстрые unit-тесты:

```bash
pytest -m unit
```

Integration-тесты, если они будут добавлены отдельно и безопасно настроены:

```bash
pytest -m integration
```

## Что нельзя включать в startup-проверку

В startup-проверку приложения запрещено включать тесты, которые:

- делают реальные запросы в OpenRouter или другие сетевые API;
- отправляют сообщения или изображения в настоящий Telegram-канал;
- используют реальные секреты, production-БД или постоянные пользовательские данные;
- требуют ручного подтверждения, долгого ожидания, бесконечного scheduler/main-цикла или зависят от порядка запуска;
- помечены `@pytest.mark.integration`.
