# Задача 08. Правила работы агентов над модулями

## Цель

Зафиксировать единые правила, чтобы разные агенты могли независимо выполнять задачи по модулям и не ломали интерфейсы друг друга.

## Общий порядок работы агента

1. Прочитать `doc`.
2. Прочитать `docs/ARCHITECTURE.md`.
3. Прочитать свой файл задачи в `docs/tasks/`.
4. Проверить существующие файлы модуля.
5. Реализовать только свой модуль и необходимые тесты.
6. Не менять публичные контракты без обновления `docs/ARCHITECTURE.md` и связанных задач.
7. Запустить релевантные тесты.
8. Обновить документацию своего модуля в `docs/modules/<module>.md`.

## Обязательная структура итоговой документации модуля

```markdown
# Модуль <name>

## Назначение

## Публичные классы и функции

## Используемые настройки

## Взаимодействие с другими модулями

## Обработка ошибок

## Тестирование

## Пример использования
```

## Правила интерфейсов

1. Общие сущности импортировать из `app.schemas`.
2. Настройки получать через `app.config.get_settings()` или принимать объект `Settings` снаружи.
3. В тестах использовать fake/mock, а не реальные внешние сервисы.
4. `service.py` должен оставаться единственным местом бизнес-процесса публикации.
5. `scheduler.py` должен только запускать job по расписанию.
6. `main.py` должен только собирать зависимости и запускать приложение.

## Что нельзя делать

- Не добавлять реальные секреты в репозиторий.
- Не делать реальные API-вызовы в unit-тестах.
- Не смешивать ответственность модулей.
- Не менять имена публичных методов без согласования документации.
- Не оставлять модуль без `docs/modules/<module>.md`.
